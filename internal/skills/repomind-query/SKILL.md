# RepoMind 编码前/问答分析

**任何涉及代码、业务逻辑、项目结构的提问，都必须先执行本流程。** 不要在未查询知识库的情况下直接回答。

## 步骤 1：读取模块索引

读取 `.repomind/index.json`，获取所有已登记的业务模块概览（描述、关键词）。

## 步骤 2：理解用户意图并扩写

- 用户真正想问什么？（代码位置？业务逻辑？调用链？）
- 涉及哪些业务领域或模块？
- 可能影响的上下游是什么？

## 步骤 2.5：业务概念卡片（concepts 目录查卡）

**关键新步骤**。在提取关键词之前，先读取 `.repomind/concepts/` 目录中的业务概念卡片：

1. 读取 `.repomind/concepts/README.md`，确认这是”业务概念卡片”目录
2. 浏览 `.repomind/concepts/*.md`（排除 `README.md`），优先查找和用户问题最接近的概念卡片
3. 如果命中卡片，先提炼：
   - 这个概念**是什么**
   - **为什么存在**
   - 用户能感知到什么
   - 关键业务边界和易混淆概念
4. 再用这些信息辅助后续关键词提取

**默认应先看 concepts 的场景（条件放宽）：**

- **分析需求文档/产品方案**：先对齐需求中的业务概念、旧概念映射和影响范围
- **找业务 Bug / 排查业务异常**：先确认预期行为、概念边界、哪些现象是设计如此、哪些才是缺陷
- **理解业务名词/功能名词/运营名词**：哪怕用户最后是要改代码，也先看这个概念在业务上是什么意思
- **分析影响范围**：用户问“这个需求会影响哪里”“这个功能改动会波及什么”，先看业务卡片再落模块
- **比较两个概念**：如“会员和创作者等级有什么区别”
- **用户问题同时包含业务概念和代码意图**：如“Pro 角色的模型配置在哪改”

**可以不先看 concepts 的场景：**

- 纯技术问题：编译错误、空指针、依赖安装、测试失败
- 纯代码定位：某个函数/接口/文件在哪里
- 纯调用链问题：明确只问函数调用顺序或代码执行路径

**简化判断规则：**

只要问题**不是纯技术定位题**，并且里面出现了业务对象、产品能力、角色类型、会员等级、活动名、推荐名、运营术语、用户权益等概念，默认先扫一遍 `.repomind/concepts/`。

**适合优先查业务卡片的提问：**

- “Pro 角色是个什么功能？”
- “创作者等级和会员有什么区别？”
- “推荐瀑布流和聚合标签是什么关系？”
- “这个业务为什么这样设计？”
- “这份需求文档里的 Pro 权益会影响哪些模块？”
- “这个现象到底是不是 Bug，还是业务本来就这样？”

**如果 concepts 目录里查不到某个业务概念**，则标记为「待补充业务卡片」，在步骤 6 的 `new_findings` 中记录，供后续 `repomind-summary` 新建卡片。

## 步骤 3：提取关键词组

根据扩写后的意图 + 业务卡片中提炼出的概念语义，列出 **2-4 组关键词**（中英文、同义词、相关术语）。

- 如果用户问的是”业务概念是什么”，关键词优先围绕**概念名 + 业务领域术语**
- 如果用户问的是”代码在哪里改”，关键词优先围绕**模块名 + 代码术语 + 入口函数/接口**

## 步骤 4：查询知识库

### 4a：关键词匹配 index.json

用每组关键词匹配 `.repomind/index.json` 中各模块的 `keywords` 和 `description` 字段。

### 4a.5：优先使用业务卡片回答概念问题

如果用户问题明显在问“某个业务概念是什么/为什么有/和谁的区别是什么”，并且 `.repomind/concepts/` 中有对应卡片：

- 先输出卡片里的业务解释
- 再从 index.json 和 modules/ 中定位相关代码
- 不要直接退化成”这个词对应哪个字段/哪个函数”

### 4b：深入阅读模块文档

读匹配到的 `.repomind/modules/<file>`，获取详细业务上下文、关键代码、修改场景和 AI 注意事项。

### 4b.5：排查记录查档（troubles 目录）

**如果是排查类问题（数据不一致、功能没生效、Bug），必须执行此步骤。**

读取 `.repomind/troubles/README.md` 确认目录格式，然后扫描 `.repomind/troubles/*.md`（排除 README.md），
根据关键词匹配历史排查记录。如果命中了相似问题，直接引用排查路径。

**什么时候应该查 troubles：**
- **问题明确是"数据不对"、"没生效"、"报错了"、"和预期不符"** → 优先查，可能有现成排查路径
- **问题涉及两端数据对不上**（如列表 A 和列表 B 数字不同）→ 必查，常见数据类型
- **纯问业务概念是什么** → 不用查，先看 concepts 卡片

**一句话：只要你在动手 grep/查表之前，先看一眼有没有前人踩过同一坑。**

### 4c：按需使用 graphify skill

graphify 提供 concepts + modules + index 无法替代的能力：**实时从 AST 查调用链、定位函数定义、追踪跨模块依赖。**

以下场景**必须调用 graphify skill 进行查询**（注意：这是调用 graphify skill，不是直接运行 `graphify` 命令行）：

- **在 Claude Code 中**：输入 `/graphify query "<问题>"`
- **在 Codex 中**：输入 `$graphify query "<问题>"`

| 条件 | 原因 |
|------|------|
| 1. concepts 目录中找不到匹配的概念卡片 | 无业务语义基础，必须从代码结构切入 |
| 2. 模块文档中的关键代码路径不够精确（缺函数名、调用关系） | 文档只记入口，不记完整调用链 |
| 3. 问题涉及跨模块影响范围（"改 A 会影响哪些模块"） | 跨模块依赖只能从 AST 实时读 |
| 4. 用户明确问函数/接口/文件的定义位置 | graphify 符号索引直接命中 |

**跳过 graphify 的条件（同时满足）：**
- ✅ concepts 有命中卡片
- ✅ 用户只问"这个概念是什么/为什么有/和什么区别"，不涉及任何代码操作
- ✅ 你不需要查看任何源码就能回答

**一句话：只要你觉得需要看一眼源码才能回答，就先调 graphify，不要自己逐文件 grep。**

## 步骤 5：整理代码位置及含义

按以下格式输出查询结果。**这份清单必须在回答前完成。**

**检索路径表必须如实记录**：用到的搜索方式、命中了什么、为何跳过某种方式（如 grep 替代 graphify 的原因）。这既让用户了解结果来源，也是后续判断发现是否需要存到知识库的依据。

```markdown
## 知识库查询结果

### 用户意图
（一句话说清用户想知道的）

### 检索路径
（记录本次查询实际用到的搜索方式，帮助用户理解结果来源）

| 方式 | 命中？ | 说明 |
|------|--------|------|
| concepts 卡片 | ✅/❌ | 命中了哪个概念/未命中 |
| index.json + modules | ✅/❌ | 匹配到了哪些模块 |
| graphify query | ✅/❌/跳过 | 查了什么、找到了什么 |
| grep/搜索源码 | ✅/❌/跳过 | 原因：graphify 无法定位精确代码/无 graphify/其他 |
| troubles 排查记录 | ✅/❌/跳过 | 命中了哪条排查记录/未命中/非排查类问题跳过 |

### 业务卡片命中
- 概念：...
- 是什么：...
- 为什么有：...
- 关键边界：...
- 易混淆概念：...

### 关键词组
- 组1: ...

### 相关模块及代码

#### 模块：xxx (xxx.md)
- 业务描述：...
- 关键代码：
  - `路径/文件.go:行号` — 用途
- AI 注意事项：...

### 图谱补充
- 调用关系：...
```

## 步骤 6：保存查询发现

将本次查询中**超出已有知识库的新发现**（业务概念、业务规则、模块边界更新、排查记录）写入临时文件，自动触发后续知识库更新：

**自动判定规则：**
- `new_findings` 数组**不为空** → `needs_summary: true`（有新知识需要同步）
- `new_findings` 数组**为空**（所有信息已在模块文档中）→ `needs_summary: false`

```bash
cat > .repomind/.query-findings.json << 'JSONEOF'
{
  "trigger": "问答",
  "intent": "用户意图简述",
  "known_modules": ["已命中模块名"],
  "new_findings": [
    {
      "type": "new_business_card|new_business_rule|module_update|trouble_record",
      "module": "主模块名",
      "file": "concepts/xxx.md 或路径",
      "content": "发现描述"
    }
  ],
  "needs_summary": true|false
}
JSONEOF
```

> `needs_summary` **必须**根据 `new_findings` 数组是否为空自动判定，不要手动预设 `false`。
> 即使发现信息和已有知识库部分重叠，只要有任何一条全新信息，就设为 `true`。

## 步骤 7：自动更新知识库

如果 `needs_summary == true`（即 new_findings 不为空），**自动调用 repomind-summary skill** 完成知识库更新，无需等待用户指令：

```
Skill: repomind-summary
```

> 这样确保 "查询 → 发现 → 存储" 全流程闭环，新知识不会因为忘记执行总结而被遗漏。

## 步骤 8：对话中的持续发现提醒

**重要**：repomind-query 在对话开始时执行一次，但知识可能在**整个对话过程中**持续涌现。保持敏锐，但**只记代码不会告诉你的东西**：

- **业务概念定义** — 某个功能在业务上到底是什么，不是什么
- **隐性规则** — 用户确认了某个"为什么这样设计"的原因
- **历史上下文** — 对话中分析出的代码演进、Bug 修复记录
- **业务意图** — 某个边界条件的业务含义（不是"是什么"，而是"为什么这样处理"）
- **排查经验** — 特定的查询组合、判断决策链

**不要记**：调用链路、函数步骤、SQL 语句、表字段、显式 if/else 分支（AI 读代码就能拿到这些）。

**任何时候发现了超出已有知识库的业务知识**，不要等到下次编码：
1. 手动写入 `.repomind/.query-findings.json`（或追加到已有文件）
2. 执行 `Skill: repomind-summary`

不需要重新调用 repomind-query。新知识直接闭环即可。
