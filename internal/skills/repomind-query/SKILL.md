# RepoMind 编码前/问答分析

**任何涉及业务逻辑、代码修改、代码定位、项目结构、异常排查的提问，都必须先执行本流程。** 不要在未查询知识库的情况下直接回答。

纯技术问题可以跳过本 skill，例如：依赖安装、编译器报错、语言语法、通用测试命令、和项目业务无关的工具问题。

## 核心原则：先识别维度，再组合查询

用户的问题不是单选题。一个问题可能同时包含：

- **业务概念维度**：它是什么、为什么存在、边界是什么、预期行为是什么
- **代码模块维度**：代码在哪里、入口是什么、调用链是什么、要改哪些模块
- **异常排查维度**：为什么不对、为什么没生效、是不是 Bug、以前有没有同类坑

本 skill 的目标不是机械查完整个知识库，而是：

1. 识别当前问题激活了哪些维度
2. 为每个维度选择最合适的主知识源
3. 在证据不足、概念缺失、需要确认真实代码行为时，通过其他知识源补充回答
4. 把超出已有知识库的新业务知识、模块边界、排查经验写入 `.repomind/.query-findings.json`

## 知识模块边界

### concepts：业务概念卡片

目录：`.repomind/concepts/`

负责回答：
- 某个业务概念是什么
- 为什么有这个业务能力
- 用户侧能感知到什么
- 正常预期行为是什么
- 业务边界、负向规则、易混淆概念是什么

适合作为主知识源：
- 用户问“是什么/为什么/区别/边界/预期”
- 需求、产品方案、运营规则、权益、角色、活动、等级、推荐等业务词出现
- 要判断某个现象是 Bug 还是业务本来如此

不足时的补充方式：
- 卡片不存在或描述过粗 → 查 `modules/index` 和代码，反推真实业务行为，并记录待补充卡片
- 卡片说了预期但用户在排查异常 → 查 `troubles` 看历史案例
- 卡片命中但要落修改点 → 查 `modules/index`、模块文档和 graphify/source

不能承担：
- 不能代替代码定位
- 不能代替历史排查路径
- 不能把具体 SQL、字段、函数步骤当成业务概念正文

### modules/index：业务模块与关键代码

目录：`.repomind/index.json`、`.repomind/modules/`

负责回答：
- 哪些业务模块相关
- 关键入口文件、函数、接口在哪里
- 修改时常见影响范围是什么
- 哪些 AI 注意事项、跨模块约束、业务坑点需要提前知道

适合作为主知识源：
- 用户问“代码在哪/改哪里/入口是什么/影响哪些模块”
- 已经知道业务概念，需要落到实现
- 需求分析需要评估改动范围

不足时的补充方式：
- 模块文档缺业务语义 → 查 `concepts`
- 模块文档缺精确调用链、函数定义、跨模块依赖 → 查 graphify/source
- 问题带有“不生效/数据不对/报错” → 查 `troubles`

不能承担：
- 不能单独判断业务预期是否正确
- 不要把源码里 30 秒内可读到的 if/else、SQL、字段说明重复写进模块文档

### troubles：历史排查记录

目录：`.repomind/troubles/`

负责回答：
- 过去是否出现过类似现象
- 已知排查路径、判断顺序、验证方式是什么
- 哪些数据不一致、未生效、异常表现是历史坑
- 哪些结论来自真实排查经验

适合作为主知识源：
- 用户问“为什么没生效/为什么不对/数据对不上/是不是 Bug/怎么排查”
- 现象和预期不一致
- 要在动手查代码或查表前先找历史经验

不足时的补充方式：
- troubles 未命中 → 查 `concepts` 确认预期，再查 `modules/index` 和代码定位实现
- troubles 命中但案例过旧或结论可疑 → 查 graphify/source 验证当前代码是否仍一致；如果旧结论不准确，记录 `trouble_knowledge` 供 summary 修正
- troubles 命中但缺业务解释 → 查 `concepts`

不能承担：
- 不能把历史案例直接当成当前结论
- 不能替代当前代码和当前数据验证

### graphify/source：实时代码证据

来源：graphify skill、必要时 `rg`/读取源码

负责回答：
- 函数/接口/文件定义位置
- 调用链、依赖关系、跨模块影响
- 当前代码真实行为
- 从已命中的模块入口出发，沿调用链定位真正相关代码并去噪

适合作为补充知识源：
- concepts、modules 或 troubles 给出的信息不足以定位精确代码
- modules 已给出较好的文件/函数/接口位置，需要沿调用链查看上下游实现
- 用户明确问函数、接口、文件、调用链、影响范围
- 需要验证历史知识是否仍符合当前实现

不能承担：
- 不能自动解释“为什么业务要这样设计”
- 不要把临时读到的显式代码步骤全部沉淀进知识库

## 步骤 1：读取基础索引

读取 `.repomind/index.json`，获取所有已登记的业务模块概览（描述、关键词）。

如果目录存在，也读取：
- `.repomind/concepts/README.md`：确认业务概念卡片格式
- `.repomind/troubles/README.md`：确认排查记录格式

如果 README 不存在，不要中断流程；继续按目录内容和文件名做最佳努力。

## 步骤 2：识别用户意图维度

先把用户问题拆成多个可同时存在的维度，不要只归类成一种问题。

输出一份内部判断表，回答前可摘要展示：

| 维度 | 是否激活 | 判断依据 | 本次要回答什么 |
|------|----------|----------|----------------|
| 业务概念 | ✅/❌ | 命中的业务词、预期、边界、为什么 | ... |
| 代码模块 | ✅/❌ | 代码位置、修改点、接口、调用链、影响范围 | ... |
| 异常排查 | ✅/❌ | 不生效、数据不一致、报错、和预期不符 | ... |

### 维度激活规则

**业务概念维度**在以下情况激活：
- 出现业务对象、产品能力、角色类型、会员等级、活动名、推荐名、运营术语、用户权益等概念
- 用户问“是什么/为什么/区别/边界/预期/是否应该这样”
- 需求文档、产品方案、规则变更需要先对齐业务语义
- 异常排查中需要判断正常预期

**代码模块维度**在以下情况激活：
- 用户问“在哪改/入口在哪/哪个接口/哪个函数/调用链/影响哪些模块”
- 用户准备编码，或问题最终需要落到代码变更
- concepts 或 troubles 命中后仍需要验证当前实现

**异常排查维度**在以下情况激活：
- 用户描述“没生效/数据不对/报错/异常/和预期不符/是不是 Bug”
- 涉及两端数据对不上、列表数量不一致、状态流转不一致
- 需要先找历史排查路径再决定查代码还是查数据

## 步骤 3：决定查询组合和顺序

按激活维度组合查询。一个问题可以查多个知识源，但每个知识源必须服务于明确维度。

| 激活维度 | 推荐查询顺序 | 目的 |
|----------|--------------|------|
| 仅业务概念 | concepts → 必要时 modules/code | 先回答定义、预期、边界；卡片不足时从代码补全真实行为 |
| 仅代码模块 | index/modules → graphify/source | 先定位已沉淀模块；不够精确时查当前代码 |
| 仅异常排查 | troubles → concepts → index/modules → graphify/source | 先找历史路径，再确认预期，最后落代码 |
| 业务概念 + 代码模块 | concepts → index/modules → graphify/source | 先对齐业务语义，再定位实现和改动面 |
| 业务概念 + 异常排查 | concepts → troubles → index/modules | 先确认预期，再看历史坑，必要时定位实现 |
| 代码模块 + 异常排查 | troubles → index/modules → graphify/source → 必要时 concepts | 先查已知坑，再落代码；如果涉及预期再补业务语义 |
| 三个维度都有 | concepts → troubles → index/modules → graphify/source | 先定义预期，再找历史经验，最后定位当前实现 |

如果推荐顺序和实际问题不匹配，可以调整，但必须在检索路径表里说明原因。

### 补查触发条件

只有出现以下情况时才补查其他知识源：

- 主知识源未命中，无法回答当前维度
- 主知识源命中但信息过粗，只能给方向，不能给结论
- 当前答案需要同时说明“业务预期 + 当前实现 + 历史排查”
- 不同知识源之间出现冲突，需要确认哪个是当前事实、哪个是历史经验、哪个是业务意图
- 用户明确要求影响范围、修改位置、排查步骤或验证方式

### 停止条件

不要为了覆盖率机械查完整个知识库。满足已激活维度的回答目标后即可停止：

- 业务概念维度：已经能说明“是什么、为什么、正常预期、关键边界”
- 代码模块维度：已经定位到相关模块、关键入口、必要的影响范围
- 异常排查维度：已经找到历史案例，或形成可执行的排查路径，并说明仍需验证什么

### 多源冲突处理

如果 concepts、modules、troubles、当前代码之间信息不一致：

- 当前代码说明“现在实际怎么做”
- concepts 说明“业务上期望怎么理解”
- troubles 说明“过去怎么踩坑、怎么排查”
- modules 说明“已沉淀的关键入口和模块边界”

回答时不要强行合并成一个结论；要明确写出差异，并说明下一步应验证业务预期、修正文档，还是修改代码。

## 步骤 4：执行查询

### 4a：查询 concepts

当业务概念维度激活，或其他维度需要确认预期行为时，读取 `.repomind/concepts/*.md`（排除 `README.md`）。

命中卡片后提炼：
- 这个概念**是什么**
- **为什么存在**
- 用户能感知到什么
- 关键业务边界和易混淆概念
- 正常预期行为是什么

如果 concepts 目录里查不到某个业务概念：
- 不要立即结束
- 继续查 `modules/index`、模块文档、必要时 graphify/source，从真实代码和模块语境获取线索
- 在步骤 6 的 `new_findings` 中记录 `concept_knowledge`，供 `repomind-summary` 新建或补充卡片

### 4b：查询 index.json 和 modules

当代码模块维度激活，或 concepts/troubles 需要落到实现时：

1. 根据用户问题、concepts 命中内容、troubles 命中内容提取 **2-4 组关键词**
2. 用关键词匹配 `.repomind/index.json` 中各模块的 `keywords` 和 `description`
3. 读取匹配到的 `.repomind/modules/<file>`
4. 提炼关键代码入口、业务上下文、修改场景、AI 注意事项
5. 如果模块文档已经给出较好的文件/函数/接口位置，进入 **4d graphify/source**，从这些位置出发查调用链和当前实现

关键词策略：
- 概念问题转代码：围绕“概念名 + 业务领域术语 + 可能的接口/模型名”
- 排查问题转代码：围绕“异常现象 + 状态/数据对象 + 已知历史关键词”
- 纯代码定位：围绕“函数名/接口名/文件名/模块名”

模块命中后的代码定位规则：
- modules/index 是定位入口，不是最终代码证据。
- 如果 modules 已经命中具体文件、函数、接口或模块入口，不要再从全仓库盲目 grep。
- 优先用 graphify query 以“已命中的入口 + 用户问题”查询调用链、定义位置、被调用方、调用方和跨模块依赖。
- 只读取 graphify 命中的相关文件和函数附近代码，避免把同名函数、无关测试、废弃实现、生成代码混进结论。
- 只有 graphify 不可用、图谱缺失、或 graphify 无法定位时，才降级使用 `rg`；降级原因必须写入检索路径表。

### 4c：查询 troubles

当异常排查维度激活，或需要从历史案例获取业务/代码线索时，读取 `.repomind/troubles/*.md`（排除 `README.md`）。

命中后提炼：
- 相似症状是什么
- 当时的判断路径是什么
- 最后结论是什么
- 哪些检查可以复用
- 当前问题和历史案例有什么差异
- 旧结论是否仍适用于当前代码和当前数据

troubles 没命中时：
- 不要直接判断“没有历史问题”
- 继续用 `concepts` 确认预期，用 `modules/index` 和代码定位当前实现
- 如果本次排查形成了可复用路径，在步骤 6 记录 `trouble_knowledge`

troubles 命中但旧结论不准确时：
- 不要只追加新 trouble
- 在步骤 6 记录 `trouble_knowledge`，明确指出需要修正的 trouble 文件、旧结论、当前证据、新结论
- 回答用户时说明“历史记录已不适用/需要修正”的原因

### 4d：按需使用 graphify/source

graphify 提供 concepts + modules + troubles 无法替代的能力：**实时从 AST 查调用链、定位函数定义、追踪跨模块依赖。**

以下场景必须调用 graphify skill 进行查询（注意：这是调用 graphify skill，不是直接运行 `graphify` 命令行）：

- **在 Claude Code 中**：输入 `/graphify query "<问题>"`
- **在 Codex 中**：输入 `$graphify query "<问题>"`

| 条件 | 原因 |
|------|------|
| 用户明确问函数/接口/文件定义位置 | graphify 符号索引直接命中 |
| modules/index 已命中较好的文件、函数或接口入口 | 从已命中入口沿调用链去噪，比全仓库 grep 更可靠 |
| 模块文档缺函数名、调用关系、跨模块依赖 | 文档只记入口，不记完整调用链 |
| 问题涉及影响范围或跨模块联动 | 跨模块依赖需要 AST/图谱验证 |
| concepts/troubles/modules 给出旧知识，需要确认当前代码是否一致 | 当前代码是最终证据 |

graphify 查询方式：
- 带上用户原问题
- 带上 modules 命中的模块名、文件路径、函数名或接口名
- 要求 graphify 返回与该入口相关的调用链、上下游依赖和当前实现位置
- 查询后只把与用户问题相关的路径纳入回答，过滤同名无关符号、测试、mock、生成代码和历史废弃文件

可以跳过 graphify 的条件：
- 用户只问业务概念定义、原因、区别或边界
- concepts 已命中且信息足够回答
- 不需要查看源码就能完成回答

如果 graphify 不可用，使用 `rg`/读取源码替代，并在检索路径中说明。

## 步骤 5.5：把查询结果转成回答

查询不是为了打勾。查询结果必须进入最终回答、后续行动或 summary 发现，不能查完就丢。

### 查询结果的用途

| 查询来源 | 在回答中怎么用 | 不该怎么用 |
|----------|----------------|------------|
| concepts | 说明业务定义、正常预期、边界、为什么 | 不要把 concepts 当代码入口 |
| modules/index | 说明相关模块、关键入口、修改影响面、AI 注意事项 | 不要复述模块文档全文 |
| troubles | 说明历史相似问题、可复用排查路径、旧结论是否仍适用 | 不要把历史案例直接当当前结论 |
| graphify/source | 说明当前代码事实、函数/接口位置、调用关系、影响范围 | 不要把显式代码步骤沉淀成业务知识 |

### 回答生成规则

根据用户意图维度组织最终回答：

- **业务概念维度激活**：先回答“是什么/正常预期/边界”，再说明是否需要看代码或排查记录。
- **代码模块维度激活**：给出相关模块、关键入口、修改位置和影响范围；如果业务语义会影响改法，先说明业务前提。
- **异常排查维度激活**：先说明历史 trouble 是否命中，再给当前可执行排查路径；如果历史结论已过期，明确指出需要修正 trouble。
- **多个维度同时激活**：按“业务预期 → 历史经验 → 当前实现 → 建议动作”的顺序回答。

### 回答必须包含的判断

最终回答前必须完成以下判断：

- 哪些查询结果是**当前结论依据**
- 哪些查询结果只是**背景参考**
- 哪些查询结果和当前代码/其他知识源存在冲突
- 哪些代码位置来自 modules 初筛，哪些来自 graphify/source 验证
- 是否形成了需要写入 `.repomind/.query-findings.json` 的新知识

### 查询结果作为回答依据

最终回答必须以查询结果为依据，而不是把查询结果放在回答后面当附录。

- 业务结论必须能追溯到 concepts、troubles、用户确认或当前代码证据。
- 代码位置必须能追溯到 modules/index 初筛和 graphify/source 验证；如果只来自 grep，必须说明 graphify 为什么不可用或未命中。
- 排查建议必须能追溯到 troubles、当前代码证据或本次分析形成的判断路径。
- 如果查询结果不足以支撑结论，必须明确说“当前证据不足”，并给出下一步需要查什么。
- 不要在没有查询依据时直接给确定性回答。

### 不同命中情况的处理

- **知识库命中且足够回答**：直接基于命中内容回答，并简要说明来源。
- **知识库命中但不够精确**：说明已有知识给出的方向，再补查 graphify/source 或源码。
- **知识库未命中**：明确说明未命中，然后从代码或图谱补证；如果发现应沉淀的新概念、模块边界或排查经验，写入 `new_findings`。
- **知识源冲突**：不要强行合并；分别说明 concepts、troubles、modules、当前代码各自代表什么，并给出当前应采用的判断。

### 输出控制

不要把完整“知识库查询结果”机械粘给用户。根据问题复杂度选择：

- 简单问题：只输出直接答案 + 关键来源。
- 代码修改/排查问题：输出关键证据、代码位置、排查路径或修改建议。
- 复杂问题或冲突问题：输出简化版检索路径表，说明为什么采用当前结论。

## 步骤 6：保存查询发现

将本次查询中**超出已有知识库的新发现**（业务概念、业务规则、模块边界更新、排查记录）写入临时文件，自动触发后续知识库更新：

**自动判定规则：**
- `new_findings` 数组**不为空** → `needs_summary: true`（有新知识需要同步）
- `new_findings` 数组**为空**（所有信息已在模块文档中）→ `needs_summary: false`

```bash
cat > .repomind/.query-findings.json << 'JSONEOF'
{
  "trigger": "问答",
  "intent": "用户意图简述",
  "known_modules": ["已命中模块名"],
  "new_findings": [
    {
      "type": "concept_knowledge|module_knowledge|trouble_knowledge|index_knowledge",
      "module": "主模块名",
      "file": "concepts/xxx.md 或路径",
      "content": "发现描述"
    }
  ],
  "needs_summary": true|false
}
JSONEOF
```

> `needs_summary` **必须**根据 `new_findings` 数组是否为空自动判定，不要手动预设 `false`。
> 即使发现信息和已有知识库部分重叠，只要有任何一条全新信息，就设为 `true`。

## 步骤 7：自动更新知识库

如果 `needs_summary == true`（即 new_findings 不为空），**自动调用 repomind-summary skill** 完成知识库更新，无需等待用户指令：

```
Skill: repomind-summary
```

> 这样确保 "查询 → 发现 → 存储" 全流程闭环，新知识不会因为忘记执行总结而被遗漏。

## 步骤 8：对话中的持续发现提醒

**重要**：repomind-query 在对话开始时执行一次，但知识可能在**整个对话过程中**持续涌现。保持敏锐，但**只记代码不会告诉你的东西**：

- **业务概念定义** — 某个功能在业务上到底是什么，不是什么
- **隐性规则** — 用户确认了某个"为什么这样设计"的原因
- **历史上下文** — 对话中分析出的代码演进、Bug 修复记录
- **业务意图** — 某个边界条件的业务含义（不是"是什么"，而是"为什么这样处理"）
- **排查经验** — 特定的查询组合、判断决策链

**不要记**：调用链路、函数步骤、SQL 语句、表字段、显式 if/else 分支（AI 读代码就能拿到这些）。

**任何时候发现了超出已有知识库的业务知识**，不要等到下次编码：
1. 手动写入 `.repomind/.query-findings.json`（或追加到已有文件）
2. 执行 `Skill: repomind-summary`

不需要重新调用 repomind-query。新知识直接闭环即可。
