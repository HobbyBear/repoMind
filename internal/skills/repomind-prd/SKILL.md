# RepoMind PRD 业务概念提取

当有一份 PRD、需求文档或产品方案需要沉淀为业务知识时执行本流程。

**适用场景：**
- 新项目/新模块启动，PRD 评审后需要快速建立业务认知基座
- 需求变更，需要将新增/修改的业务概念同步到知识库
- 新人 onboarding，从 PRD 建立业务概念体系

## 步骤 1：获取 PRD 内容

支持以下三种输入方式（任选其一），获取后统一转为 Markdown 文本：

1. **文件路径** — 给出 PRD 文件路径（自动读取 `.md` / `.txt` / `.pdf` 文件）
2. **直接粘贴** — 用户将 PRD 内容粘贴到对话中
3. **URL** — 文档链接（自动获取网页内容）

获取到内容后确认长度，如果内容过多（>10000 字），可以请求用户确认是否继续。

## 步骤 2：识别业务概念

通读 PRD，逐段识别其中出现的**业务概念**。重点标记以下类型：

| 类型 | 说明 | 示例 |
|------|------|------|
| **业务对象** | PRD 中定义的实体/模型 | "Pro 角色"、"创作者等级"、"MCN 机构" |
| **角色/身份** | 系统中存在的用户角色 | "普通会员"、"Pro 会员"、"审核员" |
| **业务流程** | 有明确步骤的操作链路 | "退款流程"、"实名认证流程" |
| **业务规则** | 显式的约束或判断条件 | "Pro 角色有效期 30 天"、"退款需在 7 天内申请" |
| **状态/阶段** | 实体可能处于的状态 | "待审核、已通过、已驳回" |
| **易混淆概念** | PRD 中明确区分或对比的两个概念 | "Pro 角色 vs 会员等级" |

输出一份识别到的概念清单，格式如下：

```markdown
## 识别到的业务概念

| 概念 | 类型 | 来源段落 | 说明 |
|------|------|---------|------|
| Pro 角色 | 业务对象 | §2.1 | 一种高级用户身份，拥有专属模型和权益 |
| Pro 角色有效期 | 业务规则 | §2.1.3 | 购买后 30 天有效，到期自动降级 |
| 模型配置 | 业务流程 | §3.1 | Pro 用户可自定义模型参数，包括温度、风格 |
```

## 步骤 3：对齐现有知识库

读取 `.repomind/concepts/` 目录中已有的概念卡片，逐条对比：

| 结果 | 处理方式 |
|------|---------|
| **PRD 中的概念已有卡片** | 对比 PRD 描述和卡片内容，标记差异和需要补充的字段 |
| **PRD 中的概念无卡片** | 标记为待新建 |
| **PRD 中的概念和卡片描述不一致** | 标记为待更新，说明差异点 |
| **卡片中有但 PRD 未涉及的概念** | 无需处理（已有沉淀不受影响） |

**输出差异清单：**

```markdown
## 与现有知识库的差异

| 概念 | 操作 | 差异说明 |
|------|------|---------|
| Pro 角色 | 更新 | 现有卡片缺少"有效期规则"和"模型配置入口" |
| 创作者等级 | 无变化 | PRD 描述与现有卡片一致 |
| MCN 机构 | 新建 | PRD §4 新增的业务概念，现有卡片无 |
```

## 步骤 4：创建/更新概念卡片

按差异清单逐条操作。

### 4a：新建卡片

按以下模板创建 `.repomind/concepts/<kebab-name>.md`：

```markdown
# 概念：XXX

## 是什么

（从 PRD 中提取的一句话业务定义）

## 为什么有

（PRD 中提到的业务目的或背景）

## 用户可见表现

（用户能感知到的功能或界面体现）

## 核心规则

- （从 PRD 提取的规则条款，逐条列出，不做合并）

## 易混淆概念

- 不是 ...
- 区别于 ...
```

**写作原则：**
- ✅ 尽量引用 PRD 中的原文或原意，不要自己发挥
- ✅ 核心规则逐条列出，PRD 怎么说就怎么写
- ❌ 不要编造 PRD 中不存在的内容
- ❌ 不要写入代码实现细节、表结构、接口字段
- 🤔 对 PRD 描述模糊或不确定的地方，标记为 **[待确认：……]** 而非自己编造

### 4b：更新已有卡片

读取现有卡片内容，只增补 PRD 中提到的新信息：

| 卡片字段 | 更新策略 |
|---------|---------|
| 是什么 | 如果 PRD 有更精确的定义，补充修正 |
| 为什么有 | 如果 PRD 补充了新的业务背景，追加 |
| 用户可见表现 | 追加新场景 |
| 核心规则 | 追加新规则，同类规则合并表述 |
| 易混淆概念 | 追加新发现的混淆点 |

**禁止**：不允许覆盖或删除已有内容，只做增量合并。

## 步骤 5：输出变更摘要

```markdown
## RepoMind PRD 处理完成

### 输入来源
- [文件路径] 或 [粘贴] 或 [URL]
- 字数：约 XXXX 字

### 识别到的业务概念
共识别 X 个概念（新建 Y 个、更新 Z 个、无变化 N 个）

| 概念 | 操作 | 说明 |
|------|------|------|
| Pro 角色 | 更新 | 补充了有效期规则和模型配置场景 |
| MCN 机构 | 新建 | 新业务对象，见 concepts/mcn-org.md |
| 创作者等级 | 无变化 | - |

### 标记为待确认项
- Pro 角色有效期：PRD 说"到期自动降级"，但未说明降级后原权益如何处理 → **[待确认]**
- MCN 机构：PRD 未明确 MCN 和创作者的绑定关系 → **[待确认]**

### 涉及的业务模块（供 repomind-summary 参考）
- character（Pro 角色）
- payment（MCN 机构结算）
```

## 步骤 6：标记待确认项

对话中如果有对 PRD 模糊点的确认讨论，随时补充到概念卡片中。如果用户当场澄清了某个 **[待确认]**，立即更新卡片。

## 步骤 7：清理与后续

产出 `.repomind/.query-findings.json`（如有新卡片需要关联模块文档更新）：

```bash
cat > .repomind/.query-findings.json << 'JSONEOF'
{
  "trigger": "PRD 处理",
  "intent": "从 PRD 提取业务概念",
  "known_modules": ["涉及模块"],
  "new_findings": [
    {
      "type": "new_business_card|new_business_rule|module_update",
      "module": "模块名",
      "file": "concepts/xxx.md",
      "content": "发现描述"
    }
  ],
  "needs_summary": true|false
}
JSONEOF
```

如果新建或更新了卡片，并且涉及新的代码模块关联，通知用户执行 `repomind-summary` 更新模块文档和 index.json。
