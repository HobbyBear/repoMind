# RepoMind PRD 业务概念提取

## ⚠️ 触发条件（重要）

**本 skill 仅在以下场景触发：用户主动提供一份历史 PRD/需求文档/产品描述，并要求从中提取业务概念、沉淀到知识库。**

不要自动触发。以下场景**不需要**使用本 skill：

| 场景 | 为什么不需要 |
|------|-------------|
| **编码前的正常需求分析** | 接需求 → 写代码 → `repomind-summary` 会在编码后自动增量更新知识库 |
| **未来需求评审** | 本 skill 面向历史 PRD 同步；未来需求先走 `repomind-query` 理解上下文，编码后由 `repomind-summary` 闭环 |
| **对话中讨论需求** | 讨论中的新发现通过 `repomind-query` 的持续发现机制 → `repomind-summary` 闭环 |
| **新人日常学习代码** | 直接调用 `repomind-query` 查已有知识库即可 |
| **用户只是提了一个业务问题** | 那是 `repomind-query` 的工作，不是本 skill |

**一句话：只有当用户明确给了历史文档、要求你“总结成业务卡片”时，才执行本流程。**

## 核心原则

**历史 PRD 描述的是当时的业务意图，当前代码反映的是当前实际实现。本文档描述的内容默认应该能在代码中找到实现，但仍必须对照当前代码核实。**

- **PRD 和当前代码不一致时，以当前代码为准** — 卡片写的是“当前代码实际做了什么”，同时标注历史 PRD 的意图差异
- **代码中没有对应实现的内容** — 不写成稳定业务事实，标记为「当前代码未找到实现，待核对」
- **PRD skill 主要产出 concepts 候选** — 是否同步 modules/index 交给 `repomind-summary` 的 summary gate 判断
- **重复运行必须幂等** — 同一份 PRD 或等价业务规则重复处理时，不得重复追加相同内容

## 步骤 1：获取输入

用户提供 PRD 内容，支持以下形式（任选其一）：

1. **文件路径** — 自动读取 `.md` / `.txt` / `.pdf` 文件
2. **直接粘贴** — PRD 内容直接粘贴到对话中
3. **URL** — 文档链接，自动获取内容

确认字数，如果内容过多（>10000 字）可以询问用户是否继续。

## 步骤 2：识别业务概念

通读 PRD，逐段标记出现的**业务概念**。重点识别以下类型：

- **业务对象** — PRD 中定义的实体/模型
- **角色/身份** — 系统中存在的用户角色
- **业务流程** — 有明确步骤的操作链路
- **业务规则** — 显式的约束或判断条件
- **易混淆概念** — PRD 中明确区分或对比的概念

输出识别到的概念清单：

```markdown
### 识别到的业务概念

| 概念 | 类型 | 来源 | 说明 |
|------|------|------|------|
| Pro 角色 | 业务对象 | §2.1 | 高级用户身份，拥有专属模型和权益 |
| Pro 角色有效期 | 业务规则 | §2.1.3 | 购买后 30 天有效，到期自动降级 |
| 模型配置 | 业务流程 | §3.1 | Pro 用户可自定义模型参数 |
```

## 步骤 3：对齐现有知识库 + 代码对照

### 3a：对比现有概念卡片

读取 `.repomind/concepts/` 目录，逐条对比。重复运行时必须先判断已有卡片是否已经包含同义或等价内容。

| 结果 | 处理方式 |
|------|---------|
| PRD 中的概念已有卡片，且规则等价 | 标记为已存在，不修改 |
| PRD 中的概念已有卡片，但有新增业务语义 | 标记为待合并 |
| PRD 中的概念已有卡片，但与当前代码验证结果冲突 | 标记为冲突修正 |
| PRD 中的概念无卡片 | 标记为待新建 |

### 3a.5：幂等去重合并规则

每次处理历史 PRD 都必须按以下顺序去重：

1. **按概念归并**：同义概念合并到同一张卡片，不新建近义重复卡片。
   - 例：`Pro 角色`、`高级角色`、`ProCharacter` 如果指同一能力，应合并到同一张卡片。
2. **按规则主题归并**：同类规则放到同一小节，不按 PRD 文件平铺追加。
   - 例：有效期、权益、展示、审核、状态流转分别作为规则主题。
3. **按等价语义去重**：表达不同但含义相同的规则不重复写。
   - 例：“购买后 30 天有效”和“有效期为 30 天”视为同一规则。
4. **按来源去重**：同一 PRD 来源已经记录过时，不重复追加来源。
5. **按当前代码事实合并**：如果历史 PRD 说法和当前代码事实不同，保留一条当前事实，并把 PRD 差异合并进“历史 PRD 差异/待核对”说明，不写多条互相矛盾的规则。

### 来源记录规则

每张由 PRD 更新过的卡片应保留简短来源记录，便于重复运行时判断是否已处理。

建议在卡片末尾维护：

```markdown
## 来源

- 历史 PRD：《文档名》 §章节 — 本次采纳：有效期规则、权益边界
```

来源记录要求：
- 如果来源已经存在，只补充新的采纳点，不重复追加同一行
- 如果无法确定文档名，使用用户提供的文件名、URL 标题或“用户粘贴 PRD”
- 来源只记录“从哪里来”和“采纳了什么”，不要粘贴 PRD 原文长段落

### 3b：对照代码验证

**目的不是读代码实现细节，而是用代码验证卡片描述的业务规则是否精确。** 做如下区分：

| 需要从代码提取到卡片（业务知识） | 不需要写在卡片里（代码实现） |
|---|---|
| 触发时机（实时/定时/条件触发） | if/else 分支、switch 逻辑 |
| 用户侧效果（用户能感知什么） | 具体 SQL、表字段名、函数名 |
| 数据来源（谁产的、谁提供的） | 调用链、缓存策略 |
| 数据加工链路（数据如何被处理） | struct 定义、参数传递 |

**执行方式：**

1. **查 modules/ 模块文档** — 定位关键代码文件
2. **读关键源码验证核心业务描述** — 找 3-5 个关键问题的答案：
   - 触发时机：这功能是实时生效还是定时任务？调用方是谁？
   - 数据来源：数据是用户产生的？BI 提供的？运营配置的？
   - 用户感知：用户在前台能看到什么？行为路径如何？
   - 边界条件：有没有开关控制？有没有保护逻辑？
3. **将验证结果补充到卡片** — 使卡片从"PRD 的概念描述"变成"业务实际实现描述"

代码对照时优先复用 RepoMind 现有路由：
- 先查 `.repomind/concepts/` 和 `.repomind/modules/`
- 模块文档不够精确时，再用 graphify/source 定位当前实现
- 不要在本 skill 内沉淀模块文档和 index；这些交给 `repomind-summary` 的 summary gate 判断

**不需要对每个概念都读源码。** 如果现有卡片已经清晰说明了上述维度，且代码验证后发现与描述一致，直接跳过。但如果概念在 PRD 中表述模糊（如"自动打标"、"智能推荐"），必须读源码确认真实行为。

**关键规则：PRD 和当前代码不一致时，以当前代码为准。** 直接按当前代码实际写卡片，同时记录历史 PRD 原本想要什么。

| 情况 | 卡片怎么写 |
|------|-----------|
| PRD 说 X，当前代码做 X | 写 X，一致 |
| PRD 说 X，当前代码做 Y | 写 Y（以当前代码为准），附注"PRD 描述为 X，当前实现为 Y" |
| PRD 说 X，当前代码中找不到对应实现 | 标记“当前代码未找到实现，待核对”，不写成稳定事实 |

输出对照结果清单：

```markdown
### 与现有知识库 + 代码对照结果

| 概念 | 操作 | 说明 |
|------|------|------|
| Pro 角色 | 合并 | 补充有效期规则；代码中对应 character 模块 |
| 创作者等级 | 已存在 | PRD 描述与现有卡片等价，不修改 |
| MCN 机构 | 待核对 | concepts 中无此概念，当前代码未找到实现 |
| 退款流程 | 冲突修正 | PRD 说新增部分退款，当前代码以 refund 模块实现为准 |
```

## 步骤 4：创建/更新概念卡片

按对照结果逐条操作。

操作类型：
- **新建**：概念不存在，且当前代码能支撑稳定业务事实
- **合并**：概念已存在，PRD + 当前代码验证得到新增业务语义
- **已存在**：卡片已覆盖等价内容，不修改
- **冲突修正**：卡片或 PRD 与当前代码不一致，按当前代码修正，并保留历史 PRD 差异
- **待核对**：当前代码未找到实现，或证据不足，不写成稳定事实
- **跳过**：不是业务概念、无复用价值、或只是代码实现细节

### 模板

```markdown
# 概念：XXX

## 是什么

（一句话业务定义）

## 为什么有

（业务目的或背景）

## 用户侧表现

（用户在什么场景看到什么、操作路径是怎样的）

## 系统侧数据流

（数据在系统中如何产生、流转、最终表达）
- **数据来源**：谁产的？用户/BI/运营/系统？
- **处理链路**：数据经过什么步骤加工？
- **最终表达**：数据以什么形式被前端消费？

## 核心规则

（关键约束、边界条件、业务保护逻辑，按子主题分组）

## 易混淆概念

- 不是 ...
- 区别于 ...
```

**写作原则：**
- ✅ 以代码实际实现为准写卡片，PRD 描述作为参考
- ✅ PRD 和当前代码不一致时：`[与当前代码不一致：PRD 说……，实际代码是……]`
- ✅ 代码中找不到对应实现时：`[当前代码未找到实现，待核对]`
- ✅ **从代码中提取业务知识精确化卡片描述**。例如：
  - ❌ 卡片只写"AI 自动打标" → ✅ 卡片写"每小时第 1 分钟定时执行，只处理首次审核通过的公开角色，有开关控制"
  - ❌ 卡片只写"VIP 有特权" → ✅ 卡片写"基础模型 VIP 免费，但高级模型仍需付费；抽卡半价；记忆永久保留"
- ❌ 不要编造 PRD 中不存在的内容
- ❌ **不要写纯代码实现**（if/else 分支、表字段、函数调用链）
- 🤔 模糊点标记为 **[待确认]**

**卡片精确度自查（填完后检查）：**
- [ ] 卡片有没有"触发时机""数据来源""用户感知"等维度的精确描述，还是只是 PRD 原文概括？
- [ ] 如果有"自动""默认"等词，是否说明了具体触发条件？（没有 → 补上或标[待确认]）
- [ ] 写的内容是业务语义（触发时机、数据链路），还是代码实现（SQL、函数名、字段名）？（后者删掉）

### 更新已有卡片

仅增补，禁止覆盖整张卡片。当前代码对照发现的不一致按当前代码修正对应条目，并保留历史 PRD 意图差异。

更新已有卡片时必须执行：
- 先读取完整卡片
- 找到对应主题小节
- 判断新规则是否已存在或语义等价
- 等价则不改
- 新增则合并到对应主题
- 冲突则保留当前代码事实，并把历史 PRD 差异写到同一主题下的差异说明
- 最后更新 `## 来源`，但不得重复追加同一来源

## 步骤 5：输出变更摘要

```markdown
## RepoMind PRD 处理完成

### 输入来源
- [文件路径] 或 [粘贴] 或 [URL]

### 变更汇总
共识别 X 个概念：新建 A 个、合并 B 个、已存在 C 个、冲突修正 D 个、待核对 E 个、跳过 F 个

| 概念 | 操作 | 说明 |
|------|------|------|
| Pro 角色 | 合并 | 补充有效期规则，已去重来源 |
| MCN 机构 | 待核对 | 当前代码未找到实现，不写成稳定事实 |
| 创作者等级 | 已存在 | PRD 描述与现有卡片等价 |

### 去重合并结果
- 已跳过重复规则：...
- 已合并来源：...
- 冲突修正：...

### 需关注项
- **Pro 角色有效期**：PRD 说到期自动降级，当前代码未找到降级逻辑 → **[待确认]**
- **MCN 机构**：当前代码未找到对应实现 → **[当前代码未找到实现，待核对]**
```

## 步骤 6：对话中完善

用户如果对 **[待确认]** 或 **[当前代码未找到实现，待核对]** 项做出了澄清，立即更新对应卡片。

## 步骤 7：闭环 — 自动调用 repomind-summary

卡片写完后，如果新建或更新了卡片，**自动调用 `repomind-summary`** 让它通过 summary gate 判断是否有新知识需要同步到模块文档和 index.json：

```bash
cat > .repomind/.query-findings.json << 'JSONEOF'
{
  "trigger": "PRD 处理",
  "intent": "从历史 PRD 提取业务概念并对照当前代码",
  "known_modules": [],
  "new_findings": [
    {
      "type": "concept_knowledge",
      "module": "",
      "file": "concepts/xxx.md",
      "content": "从历史 PRD 提取并经当前代码对照的业务概念卡片"
    }
  ],
  "needs_summary": true
}
JSONEOF
```
